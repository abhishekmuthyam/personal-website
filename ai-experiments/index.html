AI Experiments

This entire site is the experiment: everything below was built with an AI pair programmer, and these are the honest writeups.

Experiments shipped on this site

AWS LambdaDynamoDBShipped in one session

#1 — A global game leaderboard, from zero AWS to production

Starting point: a brand-new AWS account and no backend at all. Working with an AI agent in the terminal, the flow was: create a $1 billing alarm first, design a DynamoDB table whose sort key is a zero-padded score (so "top 10" needs no sorting code), write and deploy a Node.js Lambda, expose it via a free Function URL, and wire two browser games to it — validated end-to-end with curl before the frontend ever touched it.

The interesting failure: the public URL returned 403 Forbidden despite auth being NONE. The docs example grants one permission; it actually needs two (InvokeFunctionUrl + InvokeFunction). The AI found it by isolating the layer — invoking the Lambda directly proved the function was fine, so the block had to be the URL policy — then confirmed the fix in AWS's own re:Post forums.

Takeaway: AI pair programming shines at systematic debugging — isolate the failing layer, then search precisely. The leaderboard went from idea to production in an afternoon at $0/month.

YouTube Data APIQuota engineeringCaching

#2 — Searching all of YouTube from a static site

Problem: GitHub Pages can't hold an API key secret — anything in frontend JS is public. Solution: a tiny Lambda proxy that holds the YouTube API key in an environment variable and exposes only two operations (?q=search, ?trending).

The real engineering was quota math: YouTube gives 10,000 free units/day but a single search costs 100 units — only ~100 searches/day. So every result is cached in DynamoDB with a TTL (searches 24h, trending 1h), meaning repeated queries cost zero quota. The cache table cleans itself — DynamoDB's TTL deletion is free.

Takeaway: "free tier" engineering is a real skill — the design (proxy + cache + TTL) was shaped entirely by quota and pricing constraints, and that's exactly how production systems get shaped too.

Amazon CognitoNo SDKJWTs

#3 — Site-wide login with raw REST calls (no auth library)

Cognito's docs push you toward Amplify, a heavy SDK. Instead, this site's ~250-line auth.js talks to Cognito's REST endpoint directly with fetch — SignUp, ConfirmSignUp (email code), InitiateAuth, and refresh-token renewal. The ID token is a JWT; decoding its payload client-side gives the user's name for the header greeting, and the refresh token keeps users logged in across visits.

One script tag added to any page gives it the login button, the session, and a back button — the whole thing is portable vanilla JS.

Takeaway: understanding the protocol beats importing the SDK. Doing auth "by hand" once teaches you what every auth library actually does: tokens, expiry, refresh — that knowledge transfers everywhere.

PlaywrightHeadless Chrome49 checks

#4 — Letting the AI click-test the whole site

Instead of manually checking every feature after each change, the AI wrote a Playwright script that drives a real headless Chrome against the live site: it opens the homepage, verifies every content rail populated, plays a video, opens a news story overlay, follows it to the article page, logs in with a test account, checks the leaderboards load, sweeps every page for JavaScript console errors, and even checks mobile viewport overflow — 49 assertions in about a minute.

Its first run immediately caught a real bug: article pages were missing the shared script, so they had no back button or login. Fixed and redeployed within minutes.

Takeaway: the fastest way to trust AI-built features is to have the AI also build the harness that verifies them. Tests that click like a user find what code review can't.

GitHub ActionsRSSZero backend

#5 — "Live" feeds on a static site with cron + JSON

The news and video sections feel dynamic but there's no server: GitHub Actions run on a schedule (news every 10 minutes, videos every 3 hours), fetch public RSS feeds — including YouTube's per-channel feeds, which need no API key — and commit the results as JSON files the pages fetch client-side. The site "updates itself" through git commits made by a robot.

Takeaway: cron + static JSON covers a surprising share of "we need a backend" use cases — near-realtime freshness with free hosting and zero servers to operate.

Working notes on AI-assisted engineering

What consistently works

  • Verify before shipping: every feature here was tested with curl or a headless browser before being called done — the AI testing its own work catches most mistakes.
  • Small deployable steps: billing alarm → table → function → URL → frontend. Each step confirmed before the next; failures stay small and obvious.
  • Let the AI read the errors: pasting a raw 403 log beats describing it. The gh/aws CLIs make everything scriptable, which makes everything debuggable.
  • Constraints up front: "must stay $0/month" changed the architecture (Function URLs, provisioned capacity, caching). Stating constraints early prevents rework.

What to watch out for

  • Docs drift: the AI's built-in knowledge of pricing/free-tiers can be stale — have it search and verify current numbers before decisions that cost money.
  • Silent platform flakes: GitHub Pages deploys failed or stuck several times with the change being fine. Automation should verify what's actually serving, not just that a pipeline turned green.
  • Legal lines matter: "clone Super Mario" became an original platformer — mechanics aren't protected, but names, art, and music are. A good AI collaborator flags this before you publish, not after.
