Java Interview Prep

Real questions, straight answers — core Java, collections, streams, concurrency, and Spring Boot.

Core Java

Why is String immutable in Java?

Several reasons work together: security (a String used as a file path or class name can't be changed after validation), thread safety (immutable objects can be shared freely), and caching — the JVM interns string literals in a pool, which only works if they can never change. Immutability also lets hashCode() be cached, making Strings fast HashMap keys.

What is the contract between equals() and hashCode()?

If two objects are equal according to equals(), they must return the same hashCode(). The reverse is not required — unequal objects may share a hash code (that's a collision). Break this contract and hash-based collections silently misbehave: a HashMap may store "duplicate" keys or fail to find an entry you just put in.

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User u)) return false;
    return id.equals(u.id);
}

@Override
public int hashCode() {
    return Objects.hash(id);   // same fields as equals
}

Checked vs unchecked exceptions — when do you use each?

Checked exceptions (IOException) must be declared or handled; use them when the caller can realistically recover. Unchecked exceptions (RuntimeException) signal programming errors — invalid arguments, broken state — where recovery rarely makes sense. Modern practice (and Spring) leans heavily on unchecked exceptions to avoid polluting signatures; wrap low-level checked exceptions into domain-specific runtime ones.

What's the difference between == and equals()?

== compares references — are these the same object in memory? equals() compares logical value (if overridden; the default in Object is also reference comparison). Classic trap: Integer a = 127; Integer b = 127; gives a == b true (integer cache from -128 to 127), but Integer a = 128 makes it false. Always compare wrappers and Strings with equals().

final, finally, and finalize — what's the difference?

  • final — keyword: a final variable can't be reassigned, a final method can't be overridden, a final class can't be extended.
  • finally — block that runs after try/catch whether or not an exception was thrown; use for cleanup (or better, try-with-resources).
  • finalize() — deprecated hook the GC called before reclaiming an object. Never rely on it; use AutoCloseable instead.

Collections

How does HashMap work internally?

An array of buckets. put(key, value) computes the key's hash, maps it to a bucket index, and stores an entry node there. Collisions become a linked list in that bucket; since Java 8, a bucket that exceeds 8 entries converts to a red-black tree, so worst case lookup improves from O(n) to O(log n). When entries exceed capacity × loadFactor (default 16 × 0.75), the table doubles and entries are rehashed.

ArrayList vs LinkedList — which and when?

Almost always ArrayList. It's backed by an array: O(1) random access, cache-friendly iteration, amortized O(1) append. LinkedList wins only for frequent insertion/removal at the ends via iterator — and even then ArrayDeque is usually better. In interviews, mention that LinkedList's O(1) insert claim ignores the O(n) cost of *finding* the position first.

Comparable vs Comparator?

Comparable is implemented by the class itself (compareTo) and defines its single natural order — e.g. String's alphabetical order. Comparator is an external strategy: define as many orderings as you like without touching the class.

users.sort(Comparator.comparing(User::getAge)
                     .thenComparing(User::getName));

What is fail-fast behavior in iterators?

If a collection is structurally modified while iterating (except through the iterator's own remove()), the iterator throws ConcurrentModificationException. It's detected via a modCount field — best effort, not guaranteed. Concurrent collections like ConcurrentHashMap and CopyOnWriteArrayList use fail-safe iterators that work on a snapshot or tolerate concurrent updates.

Set vs List vs Map — quick use cases

  • List — ordered, duplicates allowed: an ordered feed of items.
  • Set — uniqueness: seen IDs, deduplication. LinkedHashSet keeps insertion order, TreeSet keeps sorted order.
  • Map — key → value lookup: caches, indexes, counters (merge() is great for counting).

Streams & Java 8+

map vs flatMap?

map transforms each element one-to-one. flatMap transforms each element into a stream and flattens the result — one-to-many. Use flatMap when each element expands to a collection: e.g. list of orders → stream of all line items.

List<String> words = sentences.stream()
    .flatMap(s -> Arrays.stream(s.split(" ")))
    .toList();

Intermediate vs terminal operations?

Intermediate ops (filter, map, sorted) return a stream and are lazy — nothing runs until a terminal op (collect, forEach, count, findFirst) is invoked. This laziness enables short-circuiting: findFirst() after a filter stops processing at the first match.

How should Optional be used (and not used)?

Use it as a return type to make "no result" explicit, and consume with map/orElse/orElseThrow rather than isPresent() + get(). Don't use Optional for fields or method parameters, and never Optional.get() without checking — that just trades NullPointerException for NoSuchElementException.

What does a lambda actually compile to?

Not an anonymous inner class. The compiler emits an invokedynamic call site; at runtime the JDK's LambdaMetafactory spins a lightweight implementation of the functional interface. Practical takeaways: lambdas can only capture effectively-final locals, and this inside a lambda refers to the enclosing instance, unlike anonymous classes.

Concurrency

What does volatile guarantee — and what doesn't it?

It guarantees visibility: a write to a volatile variable is immediately visible to other threads, and it prevents instruction reordering around it. It does not make compound actions atomic — volatile int counter; counter++ is still a race (read-modify-write). For counters use AtomicInteger; for invariants spanning multiple variables use locks.

synchronized vs ReentrantLock?

synchronized is simpler, automatically released, and heavily optimized by the JVM — the default choice. ReentrantLock adds capabilities: tryLock() with timeout, interruptible waits, fairness policy, and multiple Condition queues. Reach for it only when you need those features.

How does ConcurrentHashMap achieve thread safety?

Modern versions (Java 8+) use fine-grained CAS operations and per-bin synchronization instead of the old segment locks — reads are lock-free, writes lock only the single bucket involved. It never throws ConcurrentModificationException, and atomic helpers like computeIfAbsent make check-then-act patterns safe without external locking.

Why prefer ExecutorService over raw Threads?

Thread creation is expensive and unbounded threads can exhaust memory. A pool reuses threads, bounds concurrency, gives you Futures for results and exceptions, and centralizes shutdown. Since Java 21, virtual threads (Executors.newVirtualThreadPerTaskExecutor()) make blocking-per-task style cheap for I/O-heavy workloads.

Spring Boot

How does dependency injection work in Spring?

The container scans for components (@Component, @Service, @Repository, @Controller), instantiates them as beans, and wires dependencies — preferably through constructor injection, which keeps fields final and makes classes testable without the container. If multiple beans match a type, disambiguate with @Qualifier or @Primary.

@Controller vs @RestController?

@RestController = @Controller + @ResponseBody — return values are serialized straight into the response body (JSON by default) instead of resolving to a view template. For a JSON API, always the latter.

How do you handle exceptions globally in a REST API?

@RestControllerAdvice
public class ApiExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    ResponseEntity<ApiError> notFound(OrderNotFoundException ex) {
        return ResponseEntity.status(404)
            .body(new ApiError("ORDER_NOT_FOUND", ex.getMessage()));
    }
}

One advice class maps domain exceptions to consistent HTTP responses, keeping controllers clean. Combine with @Valid on request bodies and handle MethodArgumentNotValidException for field-level validation errors.

What does @Transactional actually do?

Spring wraps the bean in a proxy; entering the method begins (or joins) a transaction, and it commits on return or rolls back on a runtime exception (checked exceptions do not roll back by default). Key gotchas: self-invocation inside the same class bypasses the proxy so the annotation is ignored, and it only works on public methods with the default proxy mode.

How does Spring Boot auto-configuration work?

@SpringBootApplication enables @EnableAutoConfiguration, which loads configuration classes listed in AutoConfiguration.imports from each jar on the classpath. Each applies conditionally — @ConditionalOnClass, @ConditionalOnMissingBean — so having H2 on the classpath configures a DataSource unless you define your own. Your beans always win over auto-configured ones.
