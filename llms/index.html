LLM Stories & Evolution

Timeline

The transition from simple statistical language models to modern general-purpose cognitive layers has been defined by key architectural breakthroughs and massive compute scaling. Here is the chronological story of how LLMs evolved.

June 2017

The Attention Paper ("Attention Is All You Need")

Google researchers publish the landmark paper introducing the **Transformer architecture**. By replacing recurrence (RNNs/LSTMs) with self-attention mechanism, the architecture allows parallelized sequence training. This single design laid the foundation for every modern LLM.

2018 - 2019

The Bifurcation: BERT vs GPT

The industry splits into two main paths:
• **Encoder-only (BERT by Google)**: Ideal for understanding, classification, and extraction.
• **Decoder-only (GPT-1 & GPT-2 by OpenAI)**: Autoregressive generators trained on language modeling tasks (predicting the next token). GPT-2 proved that zero-shot capabilities emerge simply by scaling up text datasets.

June 2020

GPT-3 & Scaling Laws

OpenAI introduces **GPT-3** with 175 billion parameters. It establishes the "Scaling Laws"—which state that model performance increases predictably with compute, dataset size, and parameter count. Few-shot in-context learning becomes standard.

November 2022

InstructGPT, RLHF & ChatGPT

By applying **Reinforcement Learning from Human Feedback (RLHF)**, developers align raw completion models (GPT-3) into conversational assistants that understand and fulfill user instructions. ChatGPT launches, triggering a global AI acceleration.

2023 - 2024

The Rise of Open Weights & Multimodality

Meta releases **LLaMA**, democratizing top-tier models and fostering a massive open-source development ecosystem. Simultaneously, models evolve to natively accept multiple inputs (text, images, audio, video) like GPT-4o, Gemini 1.5 Pro (featuring a massive 1M+ context window), and Claude 3.5 Sonnet.

2025 - Present

Reasoning Models (System 2 Thinking)

Models shift from quick intuitive outputs (System 1) to deliberate step-by-step reasoning (System 2) during inference. Reinforcement-learned chain-of-thought models (like OpenAI's o1/o3 and DeepSeek-R1) spend time "thinking" before replying, solving complex math, coding, and logical puzzles with near-expert human reliability.

AI Agent Architecture

Systems Design

While an **LLM** operates as a passive text processor (taking a prompt and returning a completion), an **AI Agent** is an active system wrapper that leverages the LLM as its "brain" to orchestrate sequences of actions to accomplish complex goals.

Agents rely on four fundamental pillars:

  • Planning: Breaking down large tasks into smaller sub-tasks. Goal-directed search and reflection.
  • Memory: Short-term memory (in-context dialogue history) and Long-term memory (vector databases, semantic lookups).
  • Tools: External APIs, calculators, search engines, databases, and code compilation environments.
  • Action Loop: Executing actions, observing results, and correcting courses dynamically.

Tools & Skills: Extending Model Capabilities

Function Calling

By default, LLMs cannot access the internet, run code, or view databases. To resolve this, developers provide models with **tools** (schemas detailing functions the model can request to execute).

1. Function Calling Protocol

Instead of returning conversational text, the model returns a **structured JSON payload** specifying the function name and its required parameters. 

{
  "name": "calculate_mortgage",
  "arguments": {
    "principal": 350000,
    "interestRate": 4.5,
    "termYears": 30
  }
}

2. What is a "Skill"?

In agent frameworks, a **Skill** is a specialized tool wrapper comprising:
• **Declarative Metadata**: Explaining when and why to use the tool.
• **Execution Script**: The sandboxed code running the real API calls.
• **Output Handlers**: Parsing raw outputs into clean text for the LLM to process.

For example, the android-cli skill in this IDE permits an AI agent to build and deploy Android builds directly by issuing structured terminal instructions automatically.

Model Context Protocol (MCP)

Open Standard

The **Model Context Protocol (MCP)**, developed by Anthropic, is an open standard designed to replace proprietary, ad-hoc integrations for AI tools.

Before MCP, every IDE, chat app, and AI assistant required custom code to hook into a database, read git logs, or compile code. MCP solves this by introducing a standardized architecture:

The Three Pillars of MCP

  • MCP Host: The consumer application (like an IDE or an AI desktop app) that wants to leverage AI capabilities.
  • MCP Client: The interface running inside the host that coordinates tool requests and sessions.
  • MCP Server: Lightweight, specialized connectors exposing **Resources** (data/files), **Prompts** (predefined instructions), and **Tools** (executable actions).

MCP Architecture

Protocol Flow

Instead of integrating a model directly with tools, MCP decouples the host client from the tool server.
